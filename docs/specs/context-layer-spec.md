# Agentified Context Layer — Architecture Spec (v1)

**Document type:** Strategic + Implementation
**Status:** Draft
**Scope:** Primitives, scoping model, agent tools, context assembly, developer API, phasing

---

## 1. Vision

Agentified is a **context intelligence layer** that assembles the right tools, messages, and memory for each agent turn — as one interconnected system, not separate features.

Every primitive exists because it makes the next `assemble()` call smarter. Memories influence which tools surface. Tool usage creates memories. Session history shapes ranking. Entity relationships connect context across all primitives. The result is a system that gets progressively more intelligent, not a bag of independent features.

The agent gets a small set of tools to read/write context. The system handles orchestration: scoping, visibility, retrieval, cross-referencing between primitives, and (in later phases) automatic inference. It works with any agent framework and any language.

## 2. Primitives

Four active primitive types, two deferred.

| Primitive | Kinds | Description | Status |
|-----------|-------|-------------|--------|
| **Message** | — | Conversation messages (user, assistant, system, tool) | Active |
| **Memory** | fact, preference, procedure, episode | Stored knowledge — extracted or agent-created | Active |
| **Relationship** | — | Edges in the context graph connecting any two entities | Active (iteration 5) |
| **Tool** | backend, client, mcp | Executable capabilities | Active |
| **Skill** | — | Procedural knowledge packages (agentskills.io) — description + instructions + referenced tools | Active (iteration 4) |
| **Knowledge** | — | Reference documents and material the agent can search | Deferred |
| **Artifact** | — | Things generated by the agent or provided by the developer | Deferred |

### Primitive Scope Availability

| Primitive | Session | Namespace | Dataset |
|-----------|---------|-----------|---------|
| Message | Y | — | — |
| Memory | Y | Y | Y (procedure only, Phase A) |
| Relationship | Y | Y | Y |
| Tool | Y | Y | Y |
| Skill | Y | Y | Y |

**Notes:**
- Messages are always session-scoped (they ARE the conversation).
- Dataset-scoped memories are limited to procedural kind in Phase A.
- Knowledge and Artifacts are deferred — knowledge ingestion (chunking, embeddings at scale) is complex infrastructure for an edge case that skills can partially cover. Artifacts lack clear definition of storage, referencing, and chunking. Both may be added in a future phase.

### Deferred Primitives

**Knowledge:** Developers needing RAG likely have dedicated solutions. Skills already provide procedural knowledge. Revisit when real usage patterns emerge.

**Artifacts:** Underspecified — what is saved, where, how referenced, how chunked? Revisit when we have concrete use cases from production agents.

## 3. Scoping Model

### Hierarchy

```
Dataset ──1:N──> Namespace ──1:N──> Session
                     |
                     └──1:N──> Namespace (sub-namespace)
```

Sub-namespace notation: `namespace("org:team:user")` is a child of `namespace("org:team")` which is a child of `namespace("org")`. Colon-delimited path, arbitrary depth.

Tools are registered per-dataset. A Session object is created via `dataset.session(id)` and holds a reference to its parent DatasetRef. `searchTools` always searches the dataset that created the Session handle.

### Visibility Rules

| Scope queried | Sees primitives from |
|---------------|---------------------|
| Dataset | Own dataset-level only |
| Namespace | Own + parent Dataset + all descendant Namespaces + all descendant Sessions |
| Session | Own + **all ancestor Namespaces** + parent Dataset |

**Ancestor chain:** A session under `namespace("org:team:user")` sees primitives from `namespace("org:team:user")`, `namespace("org:team")`, `namespace("org")`, and the parent dataset. This ensures data stored at any level in the namespace hierarchy is visible to sessions deeper in the tree.

**Key implication:** A search at namespace scope returns results from all its child sessions and sub-namespaces. A search at session scope returns results from its own session + the full ancestor namespace chain + dataset. The agent chooses scope per query.

### Examples

```
search(scope=dataset("123"))
  -> dataset-level primitives only

search(scope=dataset("123").namespace("abc"))
  -> dataset "123" level
  -> namespace "abc" level
  -> namespace "abc:*" (all descendants)
  -> all sessions under "abc" and its descendants

search(scope=dataset("123").namespace("abc:def").session("xyz"))
  -> dataset "123" level
  -> namespace "abc" level (ancestor)
  -> namespace "abc:def" level (direct parent)
  -> session "xyz" level
```

## 4. Agent Tools

Five tools the LLM can invoke. Small surface, broad capability.

### Write Tools

| Tool | Purpose | Scope constraint |
|------|---------|------------------|
| `remember` | Store a memory (fact, preference, procedure, episode) | Agent specifies: session or namespace only (not dataset) |

### Read Tools

| Tool | Purpose | Scope |
|------|---------|-------|
| `recall` | Broad semantic search across memories and messages | Respects session visibility |
| `searchTools` | Find relevant tools and skills | Respects session visibility |
| `getMessages` | Retrieve specific messages or adjacent conversation window | Session only (own conversation) |
| `traverseGraph` | Explore entity relationships — "what connects to X?" | Respects session visibility, read-only (iteration 5) |

### Tool Signatures

```typescript
// --- Write ---

remember(input: {
  content: string;
  kind?: 'fact' | 'preference' | 'procedure' | 'episode';  // default: 'fact'
  scope: 'session' | 'namespace';  // required, no dataset
}): Promise<{ memoryId: string }>;

// --- Read ---

recall(input: {
  query: string;
  scope?: 'session' | 'namespace' | 'dataset';  // default: session visibility
  types?: Array<'memory' | 'message'>;  // default: all
  limit?: number;  // default: 10
}): Promise<Array<RecallResult>>;

searchTools(input: {
  query: string;
  limit?: number;  // default: 10
}): Promise<Array<RankedTool | RankedSkill>>;
// Skills appear in searchTools results. When the agent selects a skill,
// the system injects its instructions into context and loads its referenced tools.

getMessages(input: {
  limit?: number;       // default: 20
  afterSeq?: number;    // paginate forward
  aroundSeq?: number;   // return window centered on this seq
}): Promise<Array<StoredMessage>>;

traverseGraph(input: {
  entityId: string;
  depth?: number;       // default: 1
  relationshipTypes?: string[];  // filter by edge type
}): Promise<{
  entity: Entity;
  relationships: Array<{
    type: string;
    target: Entity;
    direction: 'outgoing' | 'incoming';
  }>;
}>;
```

### What the Agent Does NOT Do

- **Create relationships** — inferred automatically by the system (Phase B). Agent reads them via `traverseGraph`.
- **Manage messages** — messages flow through `updateConversation` / `prepareStep` (developer API). Agent reads them via `getMessages`.
- **Register tools/skills** — developer registers via `register()`. Agent discovers them via `searchTools`.
- **Remember at dataset scope** — dataset-level memories are system-managed (Phase B procedural extraction).

## 5. Context Builder (Developer API)

Replaces the flat `getMessages()` from the client SDK spec. Builder pattern that assembles an optimized context window in a single HTTP call.

### API

```typescript
// Full control (with first-message preservation and summary annotation)
const ctx = await session.context
  .messages({ strategy: 'recent+summary', maxTokens: 4000, keepFirst: true })
  .recall({
    tools: { limit: 10, minSimilarity: 0.7 },
    memories: { kinds: ['fact', 'preference'], limit: 20 },
  })
  .assemble();

// Minimal — recall defaults to everything
const ctx = await session.context
  .messages({ strategy: 'recent' })
  .recall()
  .assemble();

// Messages only, no recall
const ctx = await session.context
  .messages({ strategy: 'recent', maxTokens: 2000 })
  .assemble();

// Use it
const response = await agent.generate(ctx.messages, {
  prepareStep: session.prepareStep,
});
```

### Builder Steps

| Step | Purpose | Required |
|------|---------|----------|
| `.messages(opts)` | Configure conversation retrieval strategy | Yes |
| `.recall(opts)` | Configure what primitives to retrieve alongside messages | No (skip = messages only) |
| `.assemble()` | Execute: single HTTP call to core, returns assembled context | Yes (terminal) |

`.messages()` and `.recall()` are configuration-only (synchronous). `.assemble()` is the only async step — it sends one `POST /api/v1/context` with the full config and gets back the assembled result.

**`.messages()` options:**

| Option | Type | Default | Description |
|--------|------|---------|-------------|
| `strategy` | `ContextStrategy` | `'recent'` | Message selection strategy |
| `maxTokens` | `number` | `4000` | Token budget for messages |
| `keepFirst` | `boolean` | `false` | Always include the first user message in assembled context |

Summary annotation is automatic — when the core returns `summary` + `summaryRange`, the SDK constructs an annotated assistant message and injects it. See [Chat Management](../../docs/server/chat-management.md).

### Recall Configuration

```typescript
interface RecallConfig {
  tools?: boolean | { limit?: number; minSimilarity?: number; strategy?: 'semantic' | 'keyword' };
  memories?: boolean | { kinds?: MemoryKind[]; limit?: number; minSimilarity?: number };
}
```

`true` = use defaults. `false` = skip. Object = custom config. Omit `.recall()` entirely = no recall (messages only).

When `.recall()` is called with no args, defaults to `{ tools: true, memories: true }`.

### Recall Query Signal

The recall query is **derived from the conversation**. The developer does not pass a query string — the system infers it.

**Phase A behavior:** The recall query is the **content of the last user message**, used directly as the embedding similarity query. No LLM processing, no query expansion. If no user message exists (e.g., system-only conversation), recall returns empty results.

**Phase B enhancement:** LLM-generated query expansion — summarize intent from recent conversation, extract key entities, generate multiple search queries for better coverage.

### Token Budgeting

Messages and recalled primitives are budgeted **separately**:

| Budget | Configured via | Default |
|--------|---------------|---------|
| Messages | `.messages({ maxTokens })` | 4000 |
| Recalled primitives | `.recall({ *.limit })` | Per-type limits |

The two budgets don't compete. Messages get their full budget, recalled content gets formatted into additional system messages. Token counting: char/4 heuristic for v1.

### Return Type

```typescript
interface AssembledContext {
  // Ready-to-use: conversation + recalled content injected as system messages
  messages: Array<{ role: string; content: string }>;

  // Structured breakdown (inspection/debugging)
  recalled: {
    tools: RankedTool[];
    memories: Memory[];
  };

  // Message strategy metadata (from client-sdk-spec, preserved here)
  strategyUsed: string;
  summary?: string;           // present when strategy is summary or recent+summary
  fallback?: boolean;         // true if LLM summary failed, fell back to recent

  // Counts + estimates
  tokenEstimate: number;
  conversationMessages: number;   // messages from conversation history
  totalMessages: number;          // total in session
  includedMessages: number;       // conversation + injected system messages
}
```

`ctx.messages` is ready to pass to `agent.generate()` — it already includes recalled context formatted as system messages. `ctx.recalled` gives the developer raw structured data for custom formatting or debugging.

### How It Replaces getMessages()

The client SDK spec's `session.getMessages()` becomes:

```typescript
// Before (client-sdk-spec)
const { messages } = await session.getMessages({ strategy: 'recent+summary', maxTokens: 4000 });

// After (context builder)
const { messages } = await session.context
  .messages({ strategy: 'recent+summary', maxTokens: 4000 })
  .assemble();
```

Same result for messages-only use case. But the builder extends naturally to include recalled context.

## 6. Developer API (Non-Agent)

Methods the developer calls in their application code, not exposed to the LLM.

### Conversation Management

| Method | Purpose |
|--------|---------|
| `session.updateConversation({ messages })` | Dedup + persist user messages (write) |
| `session.context.messages(...).recall(...).assemble()` | Assemble optimized context (read) |
| `session.prepareStep` | Readonly property — tool hydration + LLM message persistence |
| `session.conversation.append(messages)` | Low-level append (no dedup) |
| `session.conversation.messages(opts)` | Low-level raw history |

### Inspection / Debugging

| Method | Purpose |
|--------|---------|
| `session.getMemories(opts?)` | List memories visible from session scope |
| `namespace.getMemories(opts?)` | List memories at namespace scope |

## 7. Rust Core — New Endpoints

Extends the endpoints in [client-sdk-spec.md](./client-sdk-spec.md).

### Memory Endpoints

| Endpoint | Method | Purpose |
|----------|--------|---------|
| `POST /api/v1/memories` | POST | Store a memory (dataset, namespace, session, kind, content in body) |
| `GET /api/v1/memories` | GET | List memories (scope params: dataset, namespace?, session?) |
| `POST /api/v1/memories/search` | POST | Semantic search over memories |
| `DELETE /api/v1/memories/{id}` | DELETE | Delete a memory |

### Skill Endpoints

| Endpoint | Method | Purpose |
|----------|--------|---------|
| `POST /api/v1/datasets/{id}/skills` | POST | Register skills for dataset |
| `GET /api/v1/datasets/{id}/skills` | GET | List skills for dataset |

Skills are stored alongside tools and returned by the discover/searchTools endpoint. When a skill is activated, the response includes its `instructions` and `tools` list so the client can inject instructions into context and load referenced tools.

### Relationship Endpoints

| Endpoint | Method | Purpose |
|----------|--------|---------|
| `POST /api/v1/relationships` | POST | Create relationship (Phase A: system-only, Phase B: agent-writable) |
| `POST /api/v1/graph/traverse` | POST | Traverse graph from entity |
| `GET /api/v1/entities/{id}` | GET | Get entity with relationships |

### Context Assembly Endpoint

| Endpoint | Method | Purpose |
|----------|--------|---------|
| `POST /api/v1/context` | POST | Unified context assembly (replaces messages-only context) |

**POST /api/v1/context** request body:

```json
{
  "dataset": "agent-xyz",
  "namespace": "user-123",
  "session": "session-abc",
  "messages": {
    "strategy": "recent+summary",
    "max_tokens": 4000
  },
  "recall": {
    "tools": { "limit": 10, "min_similarity": 0.7 },
    "memories": { "kinds": ["fact", "preference"], "limit": 20 }
  }
}
```

Response:

```json
{
  "messages": [
    { "role": "system", "content": "[Recalled context] User prefers dark mode..." },
    { "role": "user", "content": "Hello" },
    { "role": "assistant", "content": "Hi there!" }
  ],
  "recalled": {
    "tools": [{ "name": "billing_lookup", "score": 0.92 }],
    "memories": [{ "id": "...", "kind": "preference", "content": "User prefers dark mode" }]
  },
  "strategy_used": "recent+summary",
  "summary": "User is building an e-commerce app...",
  "fallback": false,
  "token_estimate": 1842,
  "conversation_messages": 28,
  "total_messages": 142,
  "included_messages": 30
}
```

**Note:** This response shape **supersedes** the context response in [client-sdk-spec.md §2.5](./client-sdk-spec.md). It extends the original fields (`strategy_used`, `summary`, `fallback`, `total_messages`, `included_messages`) with the recalled breakdown and token estimate. When `recall` is omitted from the request, the `recalled` object contains empty arrays.

## 8. Storage Schema (New Tables)

Extends the schema in [client-sdk-spec.md](./client-sdk-spec.md).

```sql
-- memories table
CREATE TABLE memories (
    id TEXT PRIMARY KEY,
    dataset_id TEXT NOT NULL DEFAULT 'default',
    namespace_id TEXT,             -- NULL = dataset-scoped
    session_id TEXT,               -- NULL = namespace or dataset-scoped
    kind TEXT NOT NULL,            -- 'fact' | 'preference' | 'procedure' | 'episode'
    content TEXT NOT NULL,
    embedding BLOB,               -- vector for semantic search
    superseded_by TEXT,            -- ID of memory that replaces this one (Phase B)
    entity_refs TEXT,              -- JSON array of entity IDs (Phase B, empty in Phase A)
    created_at TEXT NOT NULL,
    updated_at TEXT NOT NULL
);
CREATE INDEX idx_memories_scope ON memories(dataset_id, namespace_id, session_id);
CREATE INDEX idx_memories_kind ON memories(kind);

-- skills table
CREATE TABLE skills (
    id TEXT PRIMARY KEY,
    dataset_id TEXT NOT NULL DEFAULT 'default',
    name TEXT NOT NULL,            -- agentskills.io name format
    description TEXT NOT NULL,
    instructions TEXT NOT NULL,    -- SKILL.md body
    tools TEXT NOT NULL,           -- JSON array of tool names
    allowed_tools TEXT,            -- JSON array, nullable (agentskills.io experimental)
    metadata TEXT,                 -- JSON key-value, nullable
    embedding BLOB,               -- vector for discovery
    created_at TEXT NOT NULL
);
CREATE INDEX idx_skills_dataset ON skills(dataset_id);

-- entities table (for graph)
CREATE TABLE entities (
    id TEXT PRIMARY KEY,
    dataset_id TEXT NOT NULL DEFAULT 'default',
    name TEXT NOT NULL,
    type TEXT,                     -- 'user' | 'org' | 'tool' | 'memory' | etc.
    metadata TEXT,                 -- JSON
    created_at TEXT NOT NULL
);
CREATE INDEX idx_entities_dataset ON entities(dataset_id);

-- relationships table
CREATE TABLE relationships (
    id TEXT PRIMARY KEY,
    dataset_id TEXT NOT NULL DEFAULT 'default',
    namespace_id TEXT,
    session_id TEXT,
    source_entity_id TEXT NOT NULL,
    target_entity_id TEXT NOT NULL,
    relationship_type TEXT NOT NULL,  -- 'about' | 'related_to' | 'supersedes' | 'extracted_from' | etc.
    metadata TEXT,                    -- JSON
    created_at TEXT NOT NULL,
    FOREIGN KEY (source_entity_id) REFERENCES entities(id),
    FOREIGN KEY (target_entity_id) REFERENCES entities(id)
);
CREATE INDEX idx_relationships_source ON relationships(source_entity_id);
CREATE INDEX idx_relationships_target ON relationships(target_entity_id);
CREATE INDEX idx_relationships_scope ON relationships(dataset_id, namespace_id, session_id);
```

### Scope Resolution in Queries

All queries that respect visibility rules use this pattern:

```sql
-- Session visibility: own session + all ancestor namespaces + parent dataset
-- For namespace "org:team:user", ancestors are: ["org:team:user", "org:team", "org"]
-- The client computes the ancestor list and passes it as parameters.
WHERE dataset_id = ?
  AND (
    (namespace_id IS NULL AND session_id IS NULL)                    -- dataset-level
    OR (namespace_id IN (?, ?, ?) AND session_id IS NULL)            -- ancestor namespace chain
    OR (namespace_id = ? AND session_id = ?)                         -- own session
  )

-- Namespace visibility: own + parent dataset + all descendants + all descendant sessions
WHERE dataset_id = ?
  AND (
    (namespace_id IS NULL AND session_id IS NULL)                    -- dataset-level
    OR (namespace_id = ? AND session_id IS NULL)                     -- own namespace
    OR (namespace_id LIKE ? || ':%' ESCAPE '\' AND session_id IS NULL)  -- descendant namespaces
    OR (namespace_id = ? AND session_id IS NOT NULL)                 -- own sessions
    OR (namespace_id LIKE ? || ':%' ESCAPE '\' AND session_id IS NOT NULL)  -- descendant sessions
  )
```

**Ancestor chain computation:** For `namespace("org:team:user")`, the ancestor list is `["org:team:user", "org:team", "org"]`. Split on `:`, build progressively. The client computes this and passes all ancestors as SQL parameters.

**Namespace ID validation:** On write, namespace IDs are validated: must match `^[a-zA-Z0-9_-]+(:[a-zA-Z0-9_-]+)*$`. This rejects `%`, `_` (SQL wildcards), and other special characters, making LIKE patterns safe. The ESCAPE clause is a defense-in-depth measure.

## 9. Phasing

See [roadmap.md](./roadmap.md) for the iteration-level breakdown. High-level phases:

### Phase A (Iterations 1-4)

- Data structures for active primitives (messages, memories, tools, skills)
- Scoping + visibility rules + sub-namespace hierarchy
- CRUD persistence in Rust core
- 5 agent tools (remember, recall, searchTools, getMessages, traverseGraph)
- Context builder API (messages + recall)
- Unified tool model (backend, client, mcp) + skills (agentskills.io)
- Basic `recall` (vector similarity search)
- Summary + recent+summary context strategies

### Phase B (Iteration 5 + beyond)

- Entities + relationships tables, graph traversal
- Automatic relationship extraction from memories
- Automatic entity extraction from messages
- Memory consolidation (supersede chain)
- Graph-aware `recall` and context assembly
- `traverseGraph` write capability for agent
- Dataset-scoped memories (all kinds, not just procedural)

### Phase C (Future)

- Knowledge primitive (evaluate Cognee or similar for ingestion)
- Artifact primitive (define storage, referencing, chunking)
- Memory migration on tool changes
- Cross-namespace inference
- Memory decay / relevance scoring

## 10. Open Design Decisions

| Topic | Current stance | Revisit when |
|-------|---------------|--------------|
| Knowledge primitive | Deferred — skills cover procedural knowledge, RAG is complex infra | When real usage patterns emerge |
| Artifact primitive | Deferred — underspecified (storage, referencing, chunking) | When concrete use cases emerge |
| Memory migration on tool change | Deferred — agent discovers staleness organically | Phase C |
| Relationship write by agent | Read-only | Phase B |
| Dataset-scoped memory kinds | Procedure only | Phase B |
| Token counting | char/4 heuristic | When accuracy matters (large contexts) |
| Recall query derivation | Last user message, direct embedding | Phase B (LLM query expansion) |
| Embedding latency on remember() | Synchronous (200-500ms) — acceptable, tool calls already involve network | Phase B (async embedding with background fill) |
